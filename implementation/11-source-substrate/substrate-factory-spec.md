# Phase 2 — Substrate Factory Specification

**Status:** Phase 2 deliverable per `PLAN.md` §3. Specification, not
implementation. Phase 2.4 will build against this spec.

**Date:** 2026-05-25

**Discipline:** Same as Phase 1. No invention. The canonical kernel
already wires SE-05 (predictive reaching + ratification) and K1
(sub-cascade promotion via fidelity) — see §2 below for the audit.
What this spec adds is the per-peer primitive vocabulary that
translates Acorn token records (per `adapter-spec.md`) into
constraint-generation candidates appropriate to each peer's axis.

---

## 1. Material correction to PLAN.md

**PLAN.md §3 Phase 2 was wrong about scope.** It said:

> "The kernel files do NOT yet include M2/M3 predictive reaching
> + ratification. We need to add this... The kernel files do NOT
> yet include K1 sub-cascade promotion via fidelity. We need to
> add this."

This was incorrect. The canonical kernel at
`implementation/kernel/field.js` and `implementation/kernel/
ct-engine.js` already implements SE-05 and K1. The mechanisms
are present and wired. See §2 below for the audit.

The reason TodoMVC's mechanisms appeared inert is that TodoMVC's
deposition (static `kind: "derived"` constraints declared at boot)
never produced the conditions that exercise SE-05 or K1: no input
novelty meant no derived-generation, no derived-generation meant
no families to track fidelity over, no families meant no
promotions; predictive constraints generated by the gap mechanism
never had matching intake to ratify because the cascade rules
were authored to match the cascade's own writeback shape, not
incoming application novelty.

**Phase 2's actual work is smaller and more specific than PLAN
named.** The corrected scope:

1. Build the substrate factory that wraps the existing kernel
   cycle with a per-peer primitive vocabulary configuration.
2. Specify per-axis primitive vocabularies that turn Acorn token
   records into derived-constraint generation candidates (M2 path
   1) and gap-divergence into predictive-constraint generation
   candidates (M2 path 2) appropriate to each peer's axis.
3. Smoke-test one peer instantiation ingesting a small Acorn token
   stream, observing the canonical kernel cycle produce derived
   constraints, predictions, ratifications, family fidelity, and
   sub-cascade promotions emergently as the peer observes the
   stream.

PLAN.md will be updated post-Phase-2 to record this scope
correction.

## 2. Audit: what the kernel already has

Confirmed by direct reading of `implementation/kernel/field.js` and
`implementation/kernel/ct-engine.js` (2026-05-25):

### F1, F2, F4, F5 (foundational)
- Seed constraint at `constraints[0]`, permanent, never evicted
  (F1).
- Delta formula `(unresolved + stale*0.5)/N` computed at field
  scope by `Field.computeScalarDelta()`, at fast scope by
  `Field.computeFastDelta()` over a recent-window, at slow scope
  via running integration `slowDelta = slowDelta*(1-SLOW_STEP) +
  cs*SLOW_STEP` (F2).
- Tick loop in CT engine advances `Field.step`, refreshes
  vector-delta, modulates substrate, regardless of input (F4).
- Trace `Trace.append` is append-only at the channel; no rollback
  paths in the kernel (F5).

### M1, M2, M3, M4, M5 (mechanism)
- Vector-delta `{scalar, fast, slow, gap}` maintained by
  `Field.refreshVectorDelta()` (M1).
- Two distinct generation paths: `Field.generate(input)` produces
  derived constraints from input novelty (M2 path 1);
  `Field.generatePredictions()` produces predictive constraints
  when `gap >= GAP_PREDICT_THRESH` (M2 path 2). They cannot be
  conflated — `generate` takes input, `generatePredictions` takes
  no input and reads field state (M2).
- `Field.ratify(idx)` performs the predictive→ratified type
  transition with weight boost (`weight = min(3.0, weight *
  PRED_WEIGHT_BOOST)`) and trace tag `tag=ratified` (M3).
- `Field.modulate()` updates fast layer (decays toward zero plus
  current deviation) and slow layer (drifts toward current
  slow-delta), both byproducts of operation (M4).
- Trace lives at the channel; both substrate connections write
  via `Trace.append`, neither consumes as command (M5).

### K1, K2, K3 (composition)
- `Field.familyFidelity` map tracks fidelity per family-type
  (K1 substrate).
- `Field.checkPromotions()` promotes family-types whose
  `fidelityOf(famType) >= SUB_CASCADE_PROMOTE_THRESH` and which
  have not yet promoted. Promotion creates a `Subcascade` with
  members and a name derived from dominant member (K1).
- Sub-cascades are addressable by name: `Field.findByName(name)`,
  `Field.detectNamed(input)` (K2 part b).
- `Field.namingPref` accumulator updates on naming events
  (K3 placeholder — documented as inert per INVARIANTS v1.2).

### SE-12, SE-13 (extensions)
- Compound generation, fidelity tracking, promotion all wired
  (SE-12).
- Storage adapter recall path wired with `Field.recallWindow`
  and `Field.recallEventLog` (SE-13).

### S1, S2, S3 (substrate)
- Field is shared, owned by neither — both ER and CT operate on
  the same `Field` object (S1).
- Substrate-independence demonstrated empirically across TodoMVC's
  three substrates (S2).
- Render and execute couple through `Field.cascadeOutput` and
  `Field.intake.records` — neither addresses the other directly
  (S3).

### What's NOT in the kernel (documented gaps)
- K2 part (a): selection-bias toward sub-cascade members on
  naming. Per `selectFromMatches`, named-ness is tagged on the
  chosen record but no rank-sensitive selection mechanism reads
  it. INVARIANTS v1.2 explicitly names this as unrealized.
- K3 operationally: `namingPref` is maintained as an accumulator
  but no selection or modulation path consumes it. INVARIANTS
  v1.2 explicitly names this as unrealized.
- Trajectory-informed selection (KERNEL §10): the trace is
  written but selection does not consult it. Roadmap step 4
  pending.

**Phase 2 does not address these documented gaps.** They are
canon-level extensions that need formal spec work, not
implementation patches. Phase 2 honors the kernel as it stands.

## 3. The factory's contract

**File (Phase 2.4 implementation):** `substrate-factory.js`

**Public surface (as built):**

> **Reconciliation note:** the original draft listed `ct: CTengine`
> and `er: ERengine` slots on the Peer interface. Per §5 (the
> CT-engine audit), peers do NOT use the canonical CT/ER engines —
> the factory constructs its own per-peer cycle that calls Field's
> public methods directly. The Peer interface below reflects what
> was actually built. Phase 3.1 added several intake-configuration
> fields (also reflected).

```
makePeer(config) -> Peer

config (Phase 2 fields, all required): {
  FieldModule:     Object              // canonical Field module
  id:              String              // peer identifier
  axis:            String              // "kind" | "vocab" | "cooccur" | "position" | "frequency" | "composer"
  primitiveVocab:  PrimitiveVocab      // see §4
}

config (Phase 3.1 intake-configuration fields, all optional):
  dimsFn:          (token, ctx) -> Coord
  tokensFn:        (token, ctx) -> [intake-token-string, ...]
  outputVar:       String              // e.g. "--kind-role"
  defaultOutput:   String              // alphabet default
  outputAlphabet:  [String, ...]
  domainRules:     [{ when: {dim: value, ...}, then: String }, ...]
  centroids:       { outputToken: centroidTemplate, ... }
  onRatify:        (constraint, field, ctx) -> patternObj | null
  onPromote:       (subcascade, idMap, field, ctx) -> patternObj | null

Peer: {
  id:                  String
  axis:                String
  field:               Field            // per-peer canonical Field instance (state isolated; methods shared with kernel)
  primitiveVocab:      PrimitiveVocab
  intakeConfigActive:  Boolean          // true if Phase 3.1 fields wired
  outputVar:           String | null
  outputAlphabet:      [String] | null

  ingest(token, extCtx?):  IngestResult  // extCtx optional in Phase 3.3 lattice wiring
  observe():               PeerObservation
  getLastOutput():         String | null
  teardown():              void
}

PeerObservation: {
  id, axis, step:           identity
  constraintsByKind:        { seed, derived, predictive, ratified, meta, compound, other }
  constraintTotal:          Int
  ratCount:                 Int
  delta:                    { scalar, fast, slow, gap }
  mod:                      { fast, slow }   // SE-03 substrate modulation
  fidelities:               { [famType]: { avg, totalFires, observations } }
  subcascades:              SubcascadeView[]
  namingPref, namedCount:   K3 surface (currently inert per INVARIANTS K3 note)
  intakeConfig:             null | { outputVar, lastOutput, defaultOutput, alphabet, outputCounts, ... }
  stats:                    { tokensIngested, derivedGenerated, predictionsGenerated, ratificationsObserved, promotionsObserved, evictionsObserved, inventionsGenerated, outputResolutions, outputCounts }
}
```

**Behavior:**

- `makePeer(config)` instantiates a fresh canonical Field + CT + ER
  with the peer's seed and primitive vocabulary plugged in.
- `peer.ingest(token)` converts the token to the peer's input
  shape (per §4 primitive vocabularies) and runs one canonical
  CT cycle (`ct.runOnce(input)`). The cycle internally calls
  `Field.generate`, `Field.generatePredictions`, `Field.ratify`,
  `Field.modulate`, `Field.refreshVectorDelta`, etc. — exactly
  the existing canonical loop. No new control flow added.
- `peer.observe()` returns a read-only snapshot (O1 compliant —
  the observer does not write to the field).

**What the factory does NOT do:**

- Does not call `Field.generate`, `Field.generatePredictions`,
  `Field.ratify`, `Field.checkPromotions`, or any other kernel
  mechanism directly. The CT engine's loop calls these.
- Does not invent a per-peer cycle. The cycle is the kernel's
  cycle.
- Does not modify the kernel. Per Phase 2 discipline, the kernel
  stays unchanged.
- Does not orchestrate peers (no supervision per F3). Each peer
  is independent.

## 4. Per-axis primitive vocabularies

This is the spec's load-bearing content. The kernel's generation
machinery is generic (`generate(input)` takes input, produces
derived constraints). Per-axis primitive vocabularies say *what
input shape the peer accepts* and *what derived constraints it
produces*.

Each primitive vocabulary defines two interfaces:

```
PrimitiveVocab: {
  tokenToInput(token):              PeerInput
    // Converts adapter's TokenIntakeRecord into the peer's
    // input shape.
    
  generateDerivedFromNovelty(input, field): Constraint[]
    // M2 path 1: when input novelty is high, what derived
    // constraints describe its features in this peer's axis?
    
  generatePredictionsFromGap(field): Constraint[]
    // M2 path 2: when vector-delta gap exceeds threshold, what
    // predictive constraints describe input-features that would
    // close the gap in this peer's axis?
    
  matches(constraint, input): Bool
    // Does this constraint match this input? Used by CT engine's
    // evaluation phase.
    
  familyType(constraint): String | null
    // What family does this constraint belong to (for K1
    // fidelity tracking)? Returns null if constraint is not in
    // a tracked family.
}
```

### 4.1 kind-peer primitive vocabulary

**axis:** lexical kind (token type from Acorn's `TokenType`)

**tokenToInput:** `{ kind: token.kind, neighborKinds: token.neighbors_pre.map(t => t.kind).concat(token.neighbors_post.map(t => t.kind)) }`

**Derived constraint primitives:**

- **kind-presence:** `{type: "kind-presence", kind: K}` — fires
  when an input has `kind == K`. Generated when input arrives
  with a kind the field has not seen in K-PRESENCE-NOVELTY-WINDOW
  recent operations.

- **kind-cooccurs-within-window:** `{type: "kind-cooccurs",
  a: K1, b: K2}` — fires when an input has kind K1 and any
  neighbor (pre or post) has kind K2. Generated when a (K1, K2)
  pair appears together for the first time, or co-occurs
  significantly more often than baseline.

- **kind-transition:** `{type: "kind-transition", from: K1,
  to: K2}` — fires when an input has kind K2 and the immediately
  preceding neighbor has kind K1. Generated when a (K1→K2)
  transition appears with frequency above transition-baseline.

- **kind-run:** `{type: "kind-run", kind: K, length: L}` — fires
  when an input has kind K and the preceding L-1 neighbors all
  have kind K. Generated when a run of length L is observed.

**Predictive constraint primitives:**

- **kind-presence-predicted:** `{type: "kind-presence",
  kind: K}` — generated when the field has many derived
  constraints over a kind-co-occurrence shape that includes K
  on one side, but K-presence alone has not been ratified.

- **kind-transition-predicted:** `{type: "kind-transition",
  from: K1, to: K2}` — generated when K1-presence and K2-presence
  are both ratified, K1-K2 co-occurrence is ratified, but the
  directed transition has not been observed.

**familyType:** returns the constraint's `type` field
(`"kind-presence"`, `"kind-cooccurs"`, etc.) — families group by
constraint type, so all `kind-presence` constraints form one
family for fidelity tracking, all `kind-cooccurs` form another,
etc. K1 promotion fires when a family's measured fidelity
exceeds threshold.

### 4.2 vocab-peer primitive vocabulary

**axis:** vocabulary content (token's actual text)

**tokenToInput:** `{ text: token.text, kind: token.kind, position_class: token.position_class, neighborTexts: token.neighbors_pre.map(t => t.text).concat(token.neighbors_post.map(t => t.text)) }`

**Derived constraint primitives:**

- **text-presence:** `{type: "text-presence", text: T}` — fires
  when input has `text == T`. Generated on first novel text or
  when recurrence count crosses a threshold (the canonical case
  the SE-11 reference uses).

- **text-cooccurs:** `{type: "text-cooccurs", a: T1, b: T2}` —
  fires when input has text T1 and any neighbor has text T2.

- **text-in-position-class:** `{type: "text-in-position",
  text: T, position: P}` — fires when input has text T and
  position_class P. This is the SE-11 reference primitive
  that surfaces "this text appears in this syntactic role."

- **text-kind-binding:** `{type: "text-kind-binding", text: T,
  kind: K}` — fires when input has text T and kind K. (Stable
  binding between vocabulary and lexical role.)

**Predictive constraint primitives:**

- **text-presence-predicted:** when texts co-occurring with T
  in a tight neighborhood are ratified but T's own presence is
  not, predict T-presence.

- **text-in-position-predicted:** when T has been ratified in
  one position class and field's gap divergence suggests
  reaching, predict T in another position class.

**familyType:** returns constraint's `type` field.

### 4.3 cooccur-peer primitive vocabulary

**axis:** co-occurrence neighborhood signature

**tokenToInput:** `{ kind: token.kind, neighborhood: hash(token.neighbors_pre.map(t => t.kind) + "|" + token.neighbors_post.map(t => t.kind)) }` — the neighborhood is summarized as a hash over the kind-sequence of neighbors.

**Derived constraint primitives:**

- **neighborhood-signature:** `{type: "neighborhood-sig",
  sig: H}` — fires when input's neighborhood-hash equals H.
  Generated when a neighborhood-signature recurs.

- **kind-in-neighborhood:** `{type: "kind-in-neighborhood",
  kind: K, sig: H}` — fires when input has kind K and
  neighborhood-hash H. Generated when this (kind, neighborhood)
  pair recurs.

- **neighborhood-symmetry:** `{type: "neighborhood-symmetry",
  sig: H}` — fires when input's pre-neighbors and post-neighbors
  have the same kind-sequence (read symmetrically). Generated
  when a token sits in a symmetric neighborhood (typical of
  paired delimiters).

**Predictive constraint primitives:**

- **neighborhood-completion-predicted:** when a partial
  neighborhood-signature is ratified frequently, predict its
  completed form.

**familyType:** returns constraint's `type` field.

### 4.4 position-peer primitive vocabulary

**axis:** syntactic position class (DECL/USE/CALLEE/ATTR/STR/KEY/TYPE/CTRL/OP/DELIM/OTHER)

**tokenToInput:** `{ position: token.position_class, text: token.text, kind: token.kind }`

**Derived constraint primitives:**

- **position-presence:** `{type: "position-presence",
  position: P}` — fires when input has position_class P.

- **position-text-binding:** `{type: "position-text-binding",
  position: P, text: T}` — fires when input has position P and
  text T. This is the load-bearing primitive for DECL-USE
  binding: a text appearing in both DECL and USE positions
  ratifies as a binding pair.

- **position-transition:** `{type: "position-transition",
  from: P1, to: P2}` — fires when input has position P2 and
  the preceding neighbor has position P1. (e.g., CALLEE→DELIM
  for the `foo(` pattern.)

**Predictive constraint primitives:**

- **decl-use-completion-predicted:** when a text appears in
  DECL position but not in USE position, predict USE-position
  occurrence of the same text.

**familyType:** returns constraint's `type` field.

### 4.5 frequency-peer primitive vocabulary

**axis:** recurrence statistics

**tokenToInput:** `{ recurrence_text: token.recurrence_text, recurrence_kind_text: token.recurrence_kind_text, kind: token.kind, text: token.text }`

**Derived constraint primitives:**

- **recurrence-bucket:** `{type: "recurrence-bucket",
  bucket: B}` — fires when input's `recurrence_text` falls
  in bucket B (e.g., singleton, rare [2-5], moderate [6-20],
  common [21-100], dominant [>100]). Generated when a
  recurrence value falls in a bucket the field has not seen
  among ratified constraints.

- **kind-recurrence-binding:** `{type: "kind-recurrence",
  kind: K, bucket: B}` — fires when input has kind K and
  recurrence-bucket B. (Different kinds have characteristic
  recurrence profiles — keywords are common, identifiers vary.)

- **rare-vs-common:** `{type: "rare-vs-common", rare: B,
  common: C}` — meta primitive over recurrence buckets.

**Predictive constraint primitives:**

- **recurrence-completion-predicted:** when most buckets have
  been observed but one is conspicuously absent, predict it.

**familyType:** returns constraint's `type` field.

## 5. The kernel modifications needed (revised after CT-engine audit)

**Material correction #2 to PLAN.md:** the CT engine's evaluation
path uses the constraint compiler + ER engine (`compiler.compileField`,
`compiler.computeInputRecord`, `er.evaluateAsync`). Matching is not
a simple per-constraint function; it compiles all constraints into
a postfix bytecode stream that the ER engine walks. **For per-axis
primitive vocabularies to use this path, each peer's primitives
would either need to compile to the existing instruction set
(adding opcodes per axis) or bypass the compiler entirely.**

Adding opcodes per axis is real algorithm-16 spec work and is the
correct long-term path. It is too large for Phase 2.

The honest Phase 2 scope: **peers evaluate CPU-side, not through
the compiler/ER path.** Peers maintain their own constraint set,
their own match function (from the peer's primitive vocabulary),
and run a simplified cycle that honors the canonical SE-05/K1
mechanisms. Phase 4 (per PLAN §3) handles GPU-resolution of peer
fields.

This is consistent with PLAN.md's original Phase 2 vs Phase 4
split:
- Phase 2: substrate factory + per-peer kernel cycles (CPU).
- Phase 4: GPU resolution layer (hoist peer fields to VRAM).

I was conflating them earlier when I wrote §5 originally.

### What the factory does, then

The substrate factory produces a `Peer` instance that contains:

- A canonical `Field` instance (the SE-05/K1 mechanisms run
  unchanged, just over per-axis constraints instead of text-only
  ones).
- A simplified per-peer cycle (not the full CT engine; the full
  CT engine assumes the compiler/ER path which peers bypass).
- The peer's primitive vocabulary, which the simplified cycle
  calls into for generation, matching, and predictive reaching.

The simplified per-peer cycle mirrors `_opInput` in ct-engine.js
but **uses the primitive vocabulary's match function directly**
rather than going through the compiler:

```
peer.ingest(token):
  Field.step++
  Field.refreshVectorDelta()
  vBefore = vector_delta_snapshot
  
  # Match against current field
  input = primitiveVocab.tokenToInput(token)
  matched = []
  for i, c in enumerate(Field.constraints):
      if primitiveVocab.matches(c, input):
          matched.push(i)
  
  # Ratify matched predictives
  ratified = []
  for idx in matched:
      c = Field.constraints[idx]
      if c.kind == "predictive" and Field.ratify(idx):
          ratified.push(c)
  
  # Generate derived from novelty
  novelty = 1.0 - matched.length / max(1, Field.constraints.length - 1)
  if novelty >= GEN_NOVELTY_THRESH:
      derived = primitiveVocab.generateDerivedFromNovelty(input, Field)
      Field.integrate(derived)
  
  # Mark used
  Field.markUsed(matched)
  Field.modulate()
  Field.refreshVectorDelta()
  
  # Predictive reaching
  if Field.gap >= GAP_PREDICT_THRESH:
      predictions = primitiveVocab.generatePredictionsFromGap(Field)
      Field.integrate(predictions)
  
  # K1: family fidelity + promotion
  for c in matched:
      Field.updateFamilyFidelity(primitiveVocab.familyType(c), vBefore, Field)
  Field.checkPromotions()
  
  # Flow discipline
  Field.evictStalePredictions()
  Field._enforceCaps()
```

**No kernel modifications required for Phase 2.** The factory
imports and uses the canonical kernel's Field, calling its public
methods (`ratify`, `integrate`, `markUsed`, `modulate`,
`refreshVectorDelta`, `evictStalePredictions`, `checkPromotions`,
`updateFamilyFidelity`) but constructing the cycle itself per-peer
instead of going through the CT engine's compiler/ER path.

This is **option C** to the original §5 question (kernel parameter
vs monkey-patch vs neither). The factory wraps Field but does not
modify it.

### What Phase 4 will need to add later

Phase 4 (GPU resolution) will need either:
- new opcodes for each peer's primitive vocabulary (compiler +
  shader work — algorithm 16 extension), OR
- a per-peer GPU shader that bypasses the canonical compiler and
  walks the peer's constraints directly (simpler but breaks
  S2 verification for that peer's resolution).

Phase 4 decides between these. Phase 2 leaves that decision for
Phase 4 because the peer cycle's behavior is observable CPU-side
first, and GPU hoisting becomes a translation problem once peer
behavior is known.

## 6. Smoke test (Phase 2.5 deliverable)

A small Node script (`phase-2-smoke.js`) that:

1. Loads the canonical kernel files.
2. Loads `corpus-adapter.js` (the Phase 2 implementation of the
   adapter spec).
3. Adapts a small snippet of JavaScript (e.g., the first 50
   lines of field.js itself — small enough to inspect, real
   enough to have structural patterns).
4. Instantiates one peer (kind-peer) via `makePeer({...})`.
5. Feeds each token through `peer.ingest(token)` in source-order.
6. Periodically calls `peer.observe()` and prints the trajectory:
   constraint counts per kind (derived, predictive, ratified),
   fidelities per family, sub-cascade promotions as they happen,
   delta vector.
7. After all tokens ingested, prints final observation.

**Falsification condition for Phase 2.5:**
- If the peer produces zero derived constraints, primitive
  vocabulary's `generateDerivedFromNovelty` is wrong.
- If the peer produces derived constraints but no predictives,
  gap may never exceed threshold (tune `GAP_PREDICT_THRESH` or
  reconsider the primitive vocabulary's coverage).
- If the peer produces predictives but no ratifications, the
  predictive constraints' `when` shape doesn't match any actual
  input shape from the adapter.
- If the peer produces ratifications but no sub-cascade
  promotions, `familyType` may be wrong or fidelity threshold
  too high.

Each is a different diagnostic. The smoke test reports which
stage of the canonical SE-05/K1 cycle does or does not happen
on real source input.

If ratifications and promotions happen, Phase 2 has earned its
deliverable: the canonical kernel cycle operating on real source
input through a configurable primitive vocabulary. Phase 3
(composer substrate) can begin.

## 7. What this spec commits to and doesn't

### Commits to (as built)
- The kernel's existing SE-05 + K1 mechanisms (no rewrite).
- ~~One small kernel addition: `primitiveVocab` opts parameter to
  `Field.reset()`...~~ **Per §5 reconciliation: no kernel addition was
  made.** The factory wraps canonical Field's public methods and
  constructs the per-peer cycle in `substrate-factory.js` itself.
- Five per-axis primitive vocabularies (§4) plus a sixth composer
  vocabulary (added in Phase 3.3), each with the same shape
  (`tokenToInput`, `generateDerivedFromNovelty`,
  `generatePredictionsFromGap`, `matches`, `familyType`). Phase 3.3b
  added per-axis cross-context pattern types.
- A substrate factory (`substrate-factory.js`) that produces
  Peer instances from primitive-vocab + (Phase 3.1) intake-config
  configurations.
- A smoke test demonstrating one peer's full SE-05/K1 cycle on
  real source input (`phase-2-smoke.js`).

### Does not commit to
- The specific primitive-vocabulary content being optimal. The
  vocabularies in §4 are first drafts. Phase 6 observation may
  surface that some primitives are too coarse or too fine.
- The recurrence buckets in §4.5 being canonical. Bucket
  boundaries are tunable.
- Per-peer kernel coefficient tuning. `GAP_PREDICT_THRESH`,
  `SUB_CASCADE_PROMOTE_THRESH`, etc. start at canonical default
  values per the discipline (no tuning to engineer outcomes).
- That five peers exhaust SE-11's reference axes. They cover the
  canon's named five; future implementations may add more.
- Cross-peer coupling. That's Phase 3 (composer substrate).
- That the kernel modification is the only kernel change ever
  needed. If Phase 6 surfaces real limits in the kernel cycle,
  those become future SE-N spec work.
