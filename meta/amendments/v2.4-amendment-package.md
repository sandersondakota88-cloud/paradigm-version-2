## 6.5. Phase 5.5: imposed-precedence cleanup

### Purpose

Act on the kindMult audit finding from Phase 5. The audit demonstrated
that selection-side bias constants produce no observable change in
primary metrics because no downstream consumer reads selectFromMatches'
ranking. The architecture has been computing a sorted ranking that
nothing consumes; the constants shaping the ranking are imposed
precedence per section 8 principle 4, with no work to do.

Phase 5.5 removes the inert layer. Selection becomes set-computation
honestly. Spec and implementation match. Wire-a-consumer (Phase 5.6
or later) becomes a clean future extension on top of an honest base.

### Scope

Six imposed-precedence constants and one dead computation:

- ratified kindMult (1.3) in selectFromMatches
- meta kindMult (1.15) in selectFromMatches
- compound kindMult (1.25) in selectFromMatches
- NAMING_WEIGHT_BONUS (1.5) in CFG, consumed in selectFromMatches
- SELECT_RECENCY_EXP (1.5) in CFG, consumed in selectFromMatches
- kindBonus (15) for ratified in _enforceCaps
- the effectiveWeight ranking computation in selectFromMatches plus
  its descending sort

After removal, selectFromMatches returns the matched set with
metadata only ({idx, kind, named}). No ranking, no ordering, no
weight composition. Downstream consumers (markUsed, updateCorrelations,
ratify) continue to operate on the set as they already do.

_enforceCaps loses the ratified bonus; eviction becomes uniform
across kinds, scored by uses + weight - recency.

### Out of scope

Wire-a-consumer. The shape of a future rank-consuming selection
mechanism (top-K, weighted draw, threshold cutoff) is not specified
in 5.5 and would be a separate phase (5.6 or folded into Phase 6
prep). 5.5 is removal of dead code; 5.6 is addition of structure
the spec already commits to.

namingPref dissolution. K3 commits to naming preference emerging
from substrate accumulation rather than being addressed as an
explicit value. The current implementation has namingPref as an
explicit accumulator read by the surface display. This is a separate
K3-letter compliance question, named in INVARIANTS as a known gap,
not fixed in 5.5.

### Spec changes

KERNEL.md section 5 is rewritten to express selection as set
computation under the current spec. Pseudocode no longer multiplies
weights or sorts. Prose explicitly notes that K2 part (a) and K3
require a rank-consuming consumer to be operationally realized,
which the current kernel does not provide.

INVARIANTS.md K2 and K3 receive implementation notes acknowledging
that selection-bias enforcement is structurally specified but
currently unrealized. The invariants themselves stay; the notes are
honest about the gap between specification and current realization.

### Implementation changes

field.js:
- selectFromMatches: drop effectiveWeight composition, drop
  pow(recencyFactor, ...), drop fastBias multiplication, drop
  namedBias multiplication, drop kindMult ladder, drop sort. Return
  matched set with {idx, kind, named} metadata.
- _enforceCaps: drop kindBonus, simplify score.
- CFG: remove NAMING_WEIGHT_BONUS, SELECT_RECENCY_EXP.

Other modules (ct-engine.js, er-engine.js, reflexive-surface.js,
storage-adapter.js): no changes. They consume chosen.idx and
chosen.length, which still work.

### Estimated scope

Approximately 80-120 lines of removals across field.js. Test
updates to confirm 115/115 still passes plus a new check that the
audit script confirms zero variant difference (since variants
become trivially identical when constants are gone).

### Success criteria

1. All 115 prior tests pass unchanged.
2. Source-level audit: zero occurrences of kindMult, kindBonus,
   NAMING_WEIGHT_BONUS, SELECT_RECENCY_EXP, or effectiveWeight in
   field.js.
3. kindmult-audit.js, when re-run, reports trivially identical
   variants (audit becomes confirmation rather than discovery).
4. KERNEL section 5 pseudocode and prose match implementation.
5. INVARIANTS K2 and K3 carry implementation notes.
6. ASCII-clean, defense stack inherited.

### Dependencies

Phase 5 complete (audit shipped). Required.

### Estimated session length

One focused session.

### Risk

None of the existing tests exercise the dead ranking path, so
removal should be transparent. The risk runs the opposite direction:
if any test was implicitly depending on a constraint kind appearing
preferentially in chosen ordering (which the audit says nothing
does), removal would expose that. Test runs verify.
```

### Phase 5.6 (potential, not scheduled): wire-a-consumer

Add a rank-consuming selection mechanism that realizes K2 part (a)
and K3 operationally. Mechanism shape is open: top-K, weighted
random draw with substrate-tension-derived weights, threshold cutoff
with substrate state determining the cutoff, or something else. Per
section 8 principle 4 (let delta decide before imposing precedence),
the weights that feed the rank should derive from substrate state
(fast/slow layer modulation) and accumulated weight history rather
than from imposed kind multipliers.

Trivial to add on top of Phase 5.5's honest base. Not scheduled.
Whether it lands as 5.6, as part of Phase 6 prep, or as a separate
SE extension depends on what behavior under Phase 5 stress conditions
reveals about whether selection rank-sensitivity is needed at the
substrate's current scale.

## PIECE 4: KERNEL.md section 5 replacement (replace lines 333-365)

```
## 5. Selection (chosen-as-set under the current spec)

```
FUNCTION select(matched, substrate, subcascades):
    IF empty(matched): RETURN []

    named_member_ids = collect_member_ids_of_recently_named_subcascades(subcascades)

    chosen = []
    FOR c IN matched:
        chosen.append({
            id    : c.id,
            kind  : c.kind,
            named : c.id IN named_member_ids
        })
    RETURN chosen
```

In the current spec, selection produces the set of matched
constraints. Downstream operations -- mark_used, update_correlations,
ratify -- act on this set without consulting any ordering. Substrate
modulation (SE-03) and naming bias (SE-01, K2) influence the field's
behavior through other paths: delta drops at naming events
(NAMING_DELTA_DROP applied directly), fast/slow layer accumulation
under SE-03 modulation, weight reinforcement on ratification
(pred_weight_boost at type transition). None of these paths is a
selection ranking.

K2 part (a) commits the architecture to a "moderate selection bias
toward the sub-cascade's members" when input addresses a sub-cascade
by name. Operationally realizing this commitment requires a
rank-consuming selection mechanism (top-K, weighted draw, threshold
cutoff). The current kernel does not specify one. K2 part (b), the
moderate delta drop, is realized.

K3 commits the slow layer to accumulating a "structural preference
for inputs that address the field's internal structure by name."
Operationally realizing this commitment requires either a consumer
of accumulated naming-preference state, or routing of naming events
through SE-03 modulation such that the preference emerges in
fast/slow layer state directly (per K3's own commitment that the
preference "is not stored as an explicit value addressed by any
component" but "emerges from substrate accumulation"). The current
kernel does not specify either path.

A future kernel revision adding rank-sensitive selection would
restore weight composition. The shape of that composition --
multiplicative, additive, threshold, monotone-only -- is open and
not pre-empted by this kernel.
```

### K2 addition (insert after K2's "Source: SE-01, bootstrap v3.")

**Implementation note (v1.2):** Part (b), the "moderate delta drop,"
is realized: NAMING_DELTA_DROP is applied directly to delta when
input addresses a sub-cascade by name. Part (a), the "moderate
selection bias toward the sub-cascade's members," is structurally

Realizing part (a) requires a rank-consuming selection mechanism

that mechanism.
```

### K3 addition (insert after K3's "Source: SE-01, SE-03.")

```
**Implementation note (v1.2):** Implementations through Phase 5.5
maintain a namingPref accumulator that updates on naming events
(slow-layer-style accumulation toward 1.0, decay toward 0.0). The
accumulator is read by the reflexive surface display and by
persistence; no selection or modulation path consumes it. The
"structural preference" K3 commits to is therefore currently inert
in two senses: (1) operationally, in that it influences no behavior,
and (2) structurally, in that K3's own letter ("not stored as an
explicit value addressed by any component") is strained by namingPref
existing as a discrete addressable accumulator. Honest realization of
K3 routes naming events through SE-03 modulation such that the
preference emerges in fast/slow layer state, with no separate
accumulator. This is structural revision beyond Phase 5.5's scope;
named here as a known gap.
```
