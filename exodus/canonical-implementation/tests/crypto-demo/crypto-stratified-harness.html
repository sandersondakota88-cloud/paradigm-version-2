CRYPTO-STRATIFIED HARNESS

SHA-256 across CSS (precomputed), JS (runtime), WGSL (runtime) :: bit-precise S2 test
This harness tests whether the architecture's S2 commitment (byte-identical resolution across substrates) survives cryptographic-class operations. The previous canon-shape harness produced 2602/2602 on integer- shaped resolution. This one introduces SHA-256 as a per-rule output. Mechanics: Each rule's THEN sets the rt slot via OP_SET_HASH_RT (opcode 0x20). The hash input is the rule's predicate bytecode prefix (MATCH_DIMs + AND-reductions, as packed u32s read in little-endian byte order). The first 4 bytes of SHA-256(predicate-bytes), packed as big-endian u32, become the rt slot value. CSS path: the generator computes the hash once at compile time. The emitted CSS rule says e.g. "--rt: 1739316594". The cascade carries the precomputed value through getComputedStyle. JS path: the in-page JS oracle implements SHA-256 (hand-rolled, FIPS 180-4) and computes the hash at runtime per matching coordinate. WGSL path: the shader implements SHA-256 in WGSL and computes the hash in-shader per matching coordinate. The architecture has no GPU oracle to verify against. The check is between three independent implementations of SHA-256 (one CPU-precomputed into a CSS rule, one CPU-runtime in JS, one GPU-runtime in WGSL). If they agree byte-identical across all 1,375 fixtures, S2 holds against cryptographic operations across the full vendor stack. If they don't, we've located a vendor-specific divergence in WGSL's integer semantics (or a bug in one of the three implementations). Serve via HTTP (file:// blocks fetch): python -m http.server 8080 (from canonical-implementation/) http://localhost:8080/tests/crypto-demo/crypto-stratified-harness.html
