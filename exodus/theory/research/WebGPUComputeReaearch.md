# Encoding computation as geometry for GPU-parallel resolution**General computation can be restructured as parallel field evaluation on the GPU** — and the technical infrastructure to do so is now mature. WebGPU compute shaders (shipping in all major browsers since mid-2025)  provide storage buffers, dynamic indexing, atomic operations, and enough control flow to implement bytecode interpreters directly in WGSL. Modern physics engines already demonstrate the pattern: constraints encoded as flat buffer data, evaluated via parallel dispatch over spatial structures like signed distance fields. Academic work on GPU-side expression evaluators, model checkers, and SAT solvers confirms that tree-structured and graph-structured computations flatten effectively into GPU storage buffers and evaluate at hundreds of millions of operations per second. The convergence of these threads points toward a viable architecture where a self-describing coordinate system (like VSF) with embedded operands and AST can be encoded in GPU buffers and resolved through parallel geometric sampling rather than sequential instruction execution.-----## WebGPU compute shaders are production-ready for non-graphics workloadsWebGPU compute reached **full cross-browser availability** in 2025: Chrome 113+ (since May 2023), Safari 26 (September 2025), and Firefox 141+ (July 2025).  The compute model dispatches 3D grids of workgroups, each containing up to **256 invocations**, with threads identified by "global_invocation_id".   Data flows through storage buffers — large, dynamically-indexed "array<T>" bindings supporting read-write access — bound via bind group layouts that map JavaScript "GPUBuffer" objects to WGSL "@group/@binding" declarations.  The memory hierarchy provides four relevant address spaces: **storage** (global read/write, gigabyte-scale), **uniform** (global read-only, ~64KB, 16-byte aligned), **workgroup** (shared across a workgroup’s threads, 16,384 bytes, synchronized via "workgroupBarrier()"),  and **private** (per-invocation registers).  Atomic operations on "atomic<u32>" and "atomic<i32>" include add, min, max, bitwise ops, exchange, and compare-exchange-weak   — sufficient for lock-free data structures. Crucially, atomics in storage buffers have **device-wide (QueueFamily) scope**, enabling cross-workgroup coordination, though cross-workgroup barriers within a single dispatch remain impossible.  Multi-pass architectures (separate compute passes) handle inter-workgroup synchronization.WGSL compiles through two major pipelines: **Tint** (Chrome/Dawn → HLSL/DXIL on Windows, MSL on macOS, SPIR-V on Linux/Android) and **Naga** (Firefox/wgpu → same backends).   As of 2025, Tint uses an IR-based architecture yielding up to **7× faster shader compilation** than its previous AST-based approach.  Microsoft’s September 2024 announcement that **SPIR-V will replace DXIL** as Direct3D’s interchange format beginning with Shader Model 7 signals convergence toward SPIR-V as the universal target. For non-graphics computation, key recent features include **subgroups** (Chrome 134, February 2025), enabling SIMD-level operations like "subgroupAdd" and "subgroupBallot" with measured **2.3–2.9× speedups** for matrix-vector multiply in Google Meet;  **indirect dispatch** via "dispatchWorkgroupsIndirect()", where a GPU-written buffer specifies workgroup counts, enabling fully GPU-driven dispatch sizing;  and **timestamp queries** (quantized to 100μs for security) for pass-level performance measurement. Buffer alignment follows strict rules: "vec3<f32>" requires **16-byte alignment** despite being 12 bytes (4 bytes padding),  struct alignment equals the maximum member alignment, and struct size rounds up to a multiple of that alignment.  The practical implication for encoding instruction tables: pack fields to minimize padding, put larger-aligned members first, and use "@align(N)" or "@size(N)" attributes for explicit control.-----## Physics engines encode constraints as flat GPU data, not executable codeThe GPU physics paradigm demonstrates exactly the pattern relevant to VSF: **rules about objects become data structures that massive parallelism samples**, rather than algorithms that execute sequentially.NVIDIA PhysX 5.x runs its entire rigid body pipeline on CUDA  (SM6.0+), storing constraints in pre-allocated GPU buffer pools  with configurable capacities for constraint memory, contact buffers (double-buffered in pinned host memory), and contact patches. Its **Direct GPU API** ("eENABLE_DIRECT_GPU_API") exposes CUDA device pointers to poses, velocities, and forces, enabling end-to-end GPU pipelines  with zero CPU readback.  PhysX’s signed distance field collision system treats SDFs as a first-class geometry type — complex non-convex shapes (gears, cams) get collision detection without convex decomposition,  because the collision “rule” is encoded as a **volumetric scalar field** that threads simply sample.  Bullet 3.x implements its entire pipeline as **~50 OpenCL kernels**: parallel radix sort for spatial hashing, sweep-and-prune broadphase, narrowphase contact generation via a kernel pipeline (separating axis → stream compaction → hull clipping → contact reduction), and a Projected Gauss-Seidel constraint solver.  Constraints are grouped into independent batches via **graph coloring** — each body appears in at most one constraint per batch — so within a batch, all constraints solve in parallel  while batches execute sequentially. XPBD (Extended Position-Based Dynamics) solvers use the same pattern at higher frequency. Macklin et al.’s key  2019 insight: **n substeps with 1 iteration each** outperforms **1 step with n iterations**  by ~100× in residual error. Constraint data uses Structure-of-Arrays layouts:  "positions[N]" (float4: xyz + inverse mass), "velocities[N]", "phases[N]", with constraints as flat index arrays — distance constraints stored as "{particleA, particleB, restLength}", contact constraints as "{bodyA, bodyB, contactPoint, normal, penetrationDepth}".The spatial data structures enabling this are themselves “computation as geometry”: **uniform grid spatial hashing** transforms O(n²) pairwise collision into O(n) spatial lookup (hash position → cell → check adjacent cells),  with GPU radix sort as the enabling primitive.  **Linear BVH construction** via Morton codes builds hierarchies in ~0.2ms for ~12K objects, with optimized stackless traversal achieving **0.25ms** for generating 73,704 collision pairs from 12,486 objects. The key optimization from Tero Karras (NVIDIA): order queries by BVH leaf index so nearby threads traverse similar tree regions, minimizing warp divergence.-----## Expression trees flatten into GPU buffers via postfix encoding and stack machinesThe most mature body of work on GPU expression evaluation comes from **GPU genetic programming (GP)**, where populations of expression trees must be serialized into flat arrays for parallel evaluation. The dominant encoding is **postfix/RPN** (Reverse Polish Notation) in a flat "uint32[]" buffer, evaluated via per-thread stack machines.Langdon and Banzhaf’s 2008 SIMD interpreter stored entire GP populations as RPN expressions in GPU global memory, achieving **895 million GP operations/second**.  By 2010, this scaled to **665 billion GP ops/second** using CUDA SIMT.  The 2025 **EvoGP** system (Wang et al., arXiv:2501.17168) introduced tensorized encoding — trees of varying structure padded to uniform tensor dimensions — achieving **140× speedup** over prior GPU GP implementations with adaptive hybrid parallelism.The standard buffer layout for GPU expression evaluation:- "expr_offsets[]: uint32" — start index of each expression in the program buffer- "program[]: uint32" — packed opcodes and operand indices in postfix order- "constants[]: float" — constant pool referenced by operand indices- "variables[]: float" — input data (row-major), one row per evaluation point- "results[]: float" — output values, one per thread- Per-thread evaluation uses a fixed-depth stack in **private memory** (registers)The critical performance insight: **same-expression, different-data parallelism** eliminates warp divergence entirely, since all threads in a warp execute the same opcode sequence. When evaluating different expressions, sorting/grouping by structure maximizes warp coherence. Alternative mitigation strategies include instruction-level batching (process one opcode at a time across the population) and multiplicative masking ("result = mask_add * (a+b) + mask_mul * (a*b)") to avoid branching.Three levels of parallelism are exploited: **data parallelism** (one expression across many data points), **population parallelism** (different expressions in different threads), and **subtree parallelism** (threads within a workgroup cooperate on independent subtrees, sharing intermediate results via shared memory).For **complex ASTs with control flow**, a SPIR-V-inspired register-based encoding proves more flexible. SPIR-V’s binary format stores each instruction as "[WordCount:16 | OpCode:16]" followed by operand words (type ID, result ID, literals).  Its key design properties — opcode determines semantics fully, word count enables skipping unknown instructions, SSA form with single-definition IDs  — translate directly to a custom GPU-evaluated instruction format.  A practical register-based encoding packs into a single "uint32": "[opcode:8 | dest:8 | src1:8 | src2:8]", supporting 256 opcodes and 256 registers per thread.-----## A tiny VM runs inside a WGSL compute shader with known constraintsWGSL has sufficient dynamic control flow to implement a bytecode interpreter: "switch" on "u32/i32" for opcode dispatch (cases must be compile-time constants  but a large switch covering all opcodes works), dynamic array indexing on storage buffers for instruction and data memory, mutable local variables for registers and program counter, and "loop" with conditional "break" for the fetch-decode-execute cycle.A concrete WGSL VM pattern:wgslstruct Instruction { opcode: u32, operand_a: u32, operand_b: u32, dest: u32 }@group(0) @binding(0) var<storage, read> program: array<Instruction>;@group(0) @binding(1) var<storage, read_write> memory: array<u32>;@compute @workgroup_size(64)fn main(@builtin(global_invocation_id) gid: vec3<u32>) {    var pc: u32 = 0u;    var regs: array<u32, 16>;    var running: bool = true;    loop {        if (!running) { break; }        let inst = program[pc];        switch inst.opcode {            case 1u: { regs[inst.dest] = regs[inst.operand_a] + regs[inst.operand_b]; pc += 1u; }            case 2u: { regs[inst.dest] = memory[regs[inst.operand_a]]; pc += 1u; }            case 3u: { memory[regs[inst.operand_a]] = regs[inst.operand_b]; pc += 1u; }            case 4u: { pc = inst.operand_a; }            case 255u: { running = false; }            default: { running = false; }        }    }}**ProtonVM** (Fumero et al., 2020) demonstrated this pattern in OpenCL with 28 opcodes, per-thread stacks in private memory, shared code in global memory, and multi-heap data in local (workgroup) memory.  **fast-csg** compiles CSG expression trees directly into GPU bytecode (PTX/SASS), achieving sub-millisecond compilation by bypassing intermediate shader languages.  The NVIDIA developer forums confirm that "nvcc" does not support computed gotos in device functions  — meaning switch-case dispatch is the only viable GPU interpreter strategy, but remains efficient when the instruction set is small and threads within a warp execute the same opcode.Key limitations for a VM inside a compute shader: **no recursion** (must use an explicit stack in storage or workgroup memory), **register file size is compile-time fixed** ("array<u32, N>" where N is constant),  and **browsers may kill long-running shaders** (implementation-dependent timeout). The maximum stack depth of 16–64 entries suffices for most expression trees. The fixed-size register file maps well to a VM with known register counts.-----## SDFs encode spatial logic as fields that GPUs sample rather than executeSigned distance fields provide the purest example of “computation as geometry.” An SDF maps any point **x** to a scalar: negative inside the boundary, positive outside, zero on the surface.  The sign is a binary classifier. The magnitude is the exact Euclidean distance.  The gradient (unit-length for true SDFs) points toward the nearest surface. This **dual encoding** — boolean classification plus continuous proximity — in a single scalar is exactly what makes SDFs powerful as computation primitives.The correspondence between SDF composition and boolean logic is exact at the sign level:|CSG Operation       |SDF Formula |Logical Analog||--------------------|------------|--------------||Union (A ∪ B)       |"min(a, b)" |OR            ||Intersection (A ∩ B)|"max(a, b)" |AND           ||Complement (¬A)     |"-a"        |NOT           ||Subtraction (A \ B) |"max(a, -b)"|AND NOT       |The sign of "min(a,b)" is negative iff either a or b is negative (inside A OR inside B). The sign of "max(a,b)" is negative iff both are negative (inside A AND inside B). **SDF evaluation with CSG operators is equivalent to evaluating boolean logic on spatial membership**, but with continuous distance information rather than mere boolean results. Smooth blending operations ("opSmoothUnion" with polynomial smooth min) extend this to fuzzy/continuous logic. For collision detection, the SDF value directly encodes constraint satisfaction: "f(x) > 0" means the constraint is satisfied (point outside obstacle), "f(x) < 0" means violation (penetration by "|f(x)|"), and "∇f(x)" provides the resolution direction. In NVIDIA Flex, rigid body collision reduces to: sample SDF at particle position → if distance < 0, project out along gradient. The “don’t penetrate” rule **becomes** the zero-isosurface geometry.GPU implementation leverages hardware texture sampling for discrete SDFs (3D textures with hardware trilinear interpolation,  "R32F" format, a **256³ grid at 32-bit = 64MB**) or evaluates analytical SDFs via pure compute (one thread per query point, completely independent — zero inter-thread communication). Ray marching advances by exactly the SDF value each step, guaranteed safe by the Lipschitz condition "|∇f| ≤ 1". Recent work extends SDFs beyond 3D geometry into **configuration spaces**: Configuration Space Distance Fields (RSS 2024) map n-dimensional joint configurations to signed distance to collision boundaries, satisfying the eikonal equation in C-space and supporting boolean composition. Neural SDFs (DeepSDF, iSDF) learn continuous distance functions via MLPs, achieving real-time collision detection with batched GPU queries.  The pattern generalizes: any domain where points can be classified as inside/outside a boundary, with meaningful distance to that boundary, admits SDF representation and GPU-parallel evaluation.-----## The “computation as spatial sampling” paradigm has deep roots and GPU-native alignmentMultiple research threads converge on treating computation as field evaluation at coordinates rather than sequential instruction execution.**Dataflow architectures** provide the theoretical foundation. In a dataflow graph, nodes fire when input tokens arrive — there is no program counter.  Modern spatial dataflow processors include Wave Computing’s 16,384-PE CGRA (clockless, executing DNN dataflow graphs), Cerebras’s Wafer-Scale Engine with distributed SRAM,  SambaNova’s Reconfigurable Dataflow Unit (no fixed ISA — programmed per model),  and significantly, **Kitsune** (arXiv:2502.18403, 2025), which implements dataflow execution on conventional GPUs by constructing spatial pipelines that pair different CTA types on streaming multiprocessors. **Texture sampling is the GPU’s native “field evaluation” operation**: hardware bilinear/trilinear interpolation is nearly free, and GPU Gems 2 documented using texture lookup tables for arbitrary function approximation. **Neural fields** (NeRF, Instant-NGP) make this explicit — they parameterize physical properties as coordinate-based neural networks where computation is literally “evaluate learned function at input coordinates.” Instant-NGP’s **multiresolution hash encoding** maps coordinates to trainable feature vectors via spatial hashing across resolution levels,  achieving **1000× speedup** over vanilla NeRF by replacing expensive global computation with fast spatial lookup plus a small MLP.  This is computation-as-geometry in production.**Cellular automata** represent the purest form of spatial computing — Rule 110 is Turing-complete via purely local spatial rules.  **Lattice Boltzmann methods** recover macroscopic Navier-Stokes physics from local spatial operations on a lattice,  with each site’s computation depending only on its immediate neighborhood — perfectly matching the GPU’s SIMT model. **Processing-in-Memory** architectures (Samsung Aquabolt-XL HBM2-PIM, UPMEM) push the “compute where data lives” concept to its logical extreme,  eliminating the von Neumann bottleneck by making computation literally spatial. **Projective Geometric Algebra (PGA)** provides a mathematical framework where all geometric transformations (rotations, reflections, translations) are sandwich products using versors. PGA’s encoding of primitives and operators in blocks of four coordinates makes it **“very suitable for fast GPU implementation”** (Leo Dorst).  The Geometric Algebra Transformer (GATr, Brehmer et al., 2023) uses PGA multivectors as token representations for E(3)-equivariant neural architectures,  while GA-Unity achieves **25%+ runtime improvement** over traditional representations in game engines. -----## GPU parallel state evaluation achieves orders-of-magnitude speedups on combinatorial problemsThe most direct evidence that “evaluate entire state spaces in parallel” works comes from GPU model checking and SAT solving.**GPUexplore** (Wijs et al., Eindhoven, versions 1.0–3.0 plus 2025 Markov extension) is the most prominent GPU model checker.  It stores state vectors as arrays of 32-bit integers  in a **lock-free hash table** partitioned into **32-integer buckets** — precisely sized so one warp fetches a bucket with a single coalesced memory access.  State exploration uses “vector groups”: for an n-process state vector, n threads are assigned to construct successors, with vector groups never spanning warps.  GPUexplore 3.0  processes up to **131 million states per second** and achieves **115× speedup** over single-core LTSmin. Edelkamp and Sulewski’s model checker translates input models to **Reverse Polish Notation** — a flat integer vector — for GPU evaluation, connecting directly to the AST-encoding work.  **TurboSAT** (NVIDIA, 2025) represents the most recent breakthrough in GPU SAT solving: it reformulates satisfiability as **binarized matrix-matrix multiplication**,  encoding the CNF formula as a sparse problem matrix where each row is a clause and columns are variables, then multiplying by a binary assignment matrix where rows are candidate assignments.  The forward pass computes clause satisfaction via matrix multiply (native to GPU tensor units), and a differentiable objective function enables gradient-guided exploration.  The hybrid architecture dispatches high-confidence partial assignments to ~100 CPU CDCL threads,  achieving **5–258× speedups** over CaDiCaL on SAT Competition 2024 benchmarks. **ParaFROST** (Osama and Wijs, TACAS 2021) parallelizes SAT inprocessing on GPU,   storing the CNF formula as a **flat byte array** of clause instances with literals in 4-byte buckets, and running parallel variable elimination, subsumption, blocked clause elimination, and a custom GPU garbage collector for clause database compaction.  It dominates CPU solvers on 58% of tested benchmarks. For constraint satisfaction, GPU implementations map constraints, variables, and domain elements to threads — each thread evaluates a different constraint or domain value concurrently.   **GPU Compact-Table propagation** (2025) assigns each CUDA block to process a single 32-bit word of variable domains, with parallel reduction within the block computing removal bitmaps. GPU local search for constraint programming achieves **17× speedups** via multi-walk parallelism (multiple independent search copies) and single-walk parallelism (parallel neighborhood evaluation). GPU branch-and-bound uses the **Integer-Vector-Matrix (IVM)** data structure — a constant-size representation for subproblem pools that regularizes the otherwise irregular B&B tree  — achieving **3.3× speedup** over conventional GPU B&B with linked lists. GPU nested Monte Carlo search for combinatorial optimization reports **420× speedups** over single CPU.  The consistent pattern: irregular tree/graph computations flatten into regular GPU buffer structures, with careful attention to warp-aligned access, lock-free hash tables, and dispatch strategies that match problem parallelism to GPU hardware parallelism.-----## Architectural alignment with encoding a self-describing coordinate system in GPU buffersThe technical evidence converges on a clear architectural template for encoding something like VSF — a self-describing coordinate system with embedded operands/AST — into GPU storage buffers for parallel geometric resolution.**Buffer architecture**: Encode the coordinate system definition (axes, ranges, SDFs) as a header struct in a uniform buffer. Store the AST/expression tree in postfix order in a "storage, read" buffer of packed "uint32" instructions. Place operand data (constants, variables, state values) in separate storage buffers. Allocate per-thread evaluation stacks either in private memory (for shallow stacks ≤16 entries) or in a pre-allocated storage buffer region indexed by "global_invocation_id".**Dispatch strategy**: For evaluating the AST at many points in the coordinate space, use data-parallel dispatch (all threads evaluate the same expression at different coordinates) for zero warp divergence. For evaluating different expressions at the same point, group expressions by structure and dispatch per group. Use indirect dispatch to let a preparation shader compute workgroup counts based on the actual number of evaluation points.  **SDF integration**: Store SDF definitions as either analytical function parameters in the instruction stream (evaluated procedurally) or as 3D texture volumes with hardware interpolation. CSG composition of SDFs — "min()" for union, "max()" for intersection, negation for complement — provides boolean logic over spatial membership with continuous distance information.  The VSF’s SDF field maps directly to a constraint satisfaction/violation metric: positive values satisfy, negative values violate, magnitude measures margin, gradient provides resolution direction.**Constraint resolution**: Follow the physics engine pattern — encode constraints as flat structs ("{operandA, operandB, threshold, type}"), batch into independent groups via graph coloring, dispatch one color group per compute pass.  XPBD’s “small steps” insight (many substeps × 1 iteration beats 1 step × many iterations)  applies to iterative constraint resolution in any domain. **Performance expectations**: GPU expression evaluation runs at **hundreds of billions of operations per second** for uniform expression evaluation (GP literature), SDF sampling at a 256³ grid evaluates in under a millisecond, and state space exploration processes **~100 million states per second** (GPUexplore).  The primary bottleneck for a VM-style evaluator is warp divergence from heterogeneous instruction streams  — mitigated by ensuring threads within warps evaluate the same instruction sequence on different data.The pattern that emerges is not speculative — it is validated independently by physics simulation (constraints as flat buffers → parallel geometric resolution), genetic programming (expression trees as postfix arrays → stack-machine evaluation), model checking (state vectors as flat integer arrays → parallel BFS exploration),  and neural fields (computation as coordinate-based field evaluation → spatial hash lookup). These are all instances of the same architectural insight: **flatten structured computation into regular GPU buffers, then dispatch massively parallel threads that each evaluate a local portion of the computation space independently**. The GPU’s native operation — evaluate a function at coordinates across a grid — is the right primitive for this paradigm.  ## ConclusionThe technical intersection described in this report reveals that encoding computation as parallel field evaluation is not a theoretical aspiration but an engineering reality across multiple domains. WebGPU’s compute pipeline provides all necessary primitives  (storage buffers with dynamic indexing, workgroup shared memory, atomics, indirect dispatch) and ships in all major browsers.  The critical architectural decisions for a system like VSF are: postfix-encode expression trees for stack-machine evaluation to minimize divergence; use SDF composition algebra ("min"/"max"/negate) as the native boolean logic layer with continuous distance margins; batch constraints by independence (graph coloring) for multi-pass parallel resolution; and exploit the same-expression-different-data parallelism pattern wherever possible. The GPU’s hardware texture interpolation, originally designed for rendering, turns out to be a general-purpose field evaluator  — and the convergence of neural fields, physics engines, and spatial computing research confirms that “sample a field at coordinates” is as fundamental a computation primitive as “fetch and execute an instruction.”## SourcesHere are all the sources from the
