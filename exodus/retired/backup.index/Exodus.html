Principles

Preface The Law Underneath the Theory

Every computation layer is instantiated by a lower-dimensional delivery mechanism that does not understand what it is bootstrapping.

This is not a new observation. It is the oldest pattern in the history of computing. We have just never named it clearly enough to see where it leads next.

16 bit flash Materializes 32 bit firmware. Does not understand what it is producing. Executes its one job and hands off.
32 bit firmware Materializes 64 bit operating space. Does not understand what it is producing. Executes its one job and hands off.
64 bit OS Materializes the application environment. Does not understand what applications will do. Executes its one job and hands off.
Current web stack Materializes Exodus. Does not understand what it is producing. Loads a page and hands off. Job done.

Each layer in this sequence is fully expressive in its own dimensionality. Every possible program that can run on 32 bit firmware fits within 32 bit address space. But the entire expressiveness of 32 bit space — every program, every state, every computation it can represent — is a single coordinate in the 64 bit space above it. The lower layer is not replaced. It becomes a point in the higher-dimensional space it instantiates.

The current web stack is not the territory. It is a coordinate in the space Exodus opens. HTTP, HTML, and JavaScript are fully expressive within their own dimensionality — every possible web application ever built fits within them. But the entire expressiveness of the web stack is consumed in the act of bootstrapping the next layer. after the handoff, the web stack is still running. It just stopped being where the interesting computation happens.

The flash does not understand the firmware. The firmware does not understand the OS. The OS does not understand the application. The web stack does not understand Exodus. But Exodus understands all of them. Because the next layer always does.

What the Current Platform Thinks Software Is

The browser was not designed for applications. It was designed for documents. HTML was a document format. JavaScript was a scripting layer added in 1995. HTTP was a request/response protocol for retrieving files. Every framework, every architectural pattern, every infrastructure primitive in the modern web stack is civilisation built on top of that accident.

Execution Lives in JavaScript. Mutable, imperative, single-threaded. Corrupts under failure. Takes the application with it.
Network The runtime boundary. Every cross-boundary operation is discrete, stateful, and failure-prone.
State Ephemeral by default. Hydration exists because server and client must be reconciled after the fact.
Crashes Unrecoverable by design. No structural recovery surface. The tab dies.
Collaboration Opt-in and expensive. Isolation is the default. Multiplayer is a feature you build.

These are not framework failures. They are the constraints of the current dimensionality. Every framework operates within them because every framework is built inside the flash storage layer. Exodus is not built inside it. Exodus is what the flash storage layer instantiates.

The Dimensionality Expansion

Each successive layer in the historical sequence is not just more of the same. It is a qualitative change in what problems are representable at all. Problems intractable in 32 bit space are trivial in 64 bit space — not because the clock is faster but because the representational space is larger. The Vessel Image is the next address space expansion. application state that is intractable to enumerate in the flat selector space of CSS is trivial in the coordinate space of the Vessel — not because the hardware is faster but because the space is higher-dimensional. The GPU makes it fast. The geometry makes it possible.

The end of Moore's Law is the end of the current dimensionality ceiling. Transistor scaling cannot continue. The industry's response has been specialised silicon — GPUs, TPUs, neuromorphic chips — all optimisations within the imperative switching paradigm. Exodus is not an optimisation within the current paradigm. It is the dimensionality expansion that opens the next one. application logic that currently requires switching becomes sampling. Sampling in a pre-defined coordinate space is not a faster version of switching. It is a different class of operation — one that the GPU pipeline was built to perform, natively, in every device already deployed.

The law is not new. The next instantiation is. The law is called Sacrificial Purpose.

The Model the Brain already Uses

Before the first transistor was switched, before the first program was written, before von Neumann put his architecture to paper — the brain had already solved the problem Exodus is now proposing to solve. Not as an approximation. Not as a workaround. as the native implementation of a computational principle that took two billion years of evolution to arrive at and that every human being carries in their skull consuming twenty watts.

The brain does not execute instructions. It does not fetch, decode, and execute operations against a mutable register file. It does not branch on conditions imperatively. None of the architectural assumptions of the von Neumann model apply to how biological intelligence actually works. What the brain does is this:

It maintains a vast, high-dimensional geometric structure — the connectome, the weighted network of synaptic relationships — that encodes everything the organism knows. Every pattern it has encountered. Every valid response to every class of input it has experienced. When a stimulus arrives, the brain does not execute a program to compute a response. It finds the region of that geometric structure that the stimulus activates, and the response emerges from the geometry of that region.

This is not execution. This is sampling.

The brain runs on twenty watts. It processes more information, in more dimensions, with more parallelism, than any computing system ever built. Not because neurons are fast — they are extraordinarily slow compared to transistors — but because the computation is geometric rather than sequential. The brain does not re-execute logic it already knows. It reads geometry that already encodes the answer. It does not waste energy re-discovering the shape of the world on every stimulus. The shape is already there. The stimulus collapses it to a response.

Every attempt to build general intelligence on top of the von Neumann model has hit the same wall — because the model is fundamentally misaligned with the class of computation intelligence requires. Neural networks are an attempt to approximate the brain's geometric computation inside the imperative paradigm. They work. But they are approximations running on the wrong substrate, requiring kilowatts to do what the brain does in twenty watts, because they are fighting the architecture they run on.

Exodus does not fight the architecture. It aligns with it. The GPU was built for exactly the class of operation the brain uses — massively parallel spatial resolution against a static geometric structure. The brain arrived at this model through evolution. Exodus arrives at it through following the theory of computation honestly to where it actually leads.

The brain is not a metaphor for Exodus. It is the existence proof. Geometric computation works at scale. It runs in every human being on earth. It has been running for two billion years. We just built our computers wrong.

Why Language Is the One Thing That Never Leaves

Every framework will be rewritten. Every runtime will be replaced. The web stack will become the bootstrapper and step back. Codebases will be recompiled toward geometry rather than toward bytecode. The targets change. The tools adapt. This is the normal motion of a paradigm shift and none of it is loss — it is the lower layer fulfilling its Sacrificial Purpose and handing off.

But programming languages are different. and the reason they are different is the most important philosophical observation in this entire document.

a programming language is not a tool for talking to computers. It is a tool for representing human thought about logic with enough precision that it can be made executable. The language came first. The computer came second. The language is the human side of the interface — the most precise representation we have ever developed of how human minds reason about cause, effect, state, and transformation.

When we write code, we are not primarily describing what a machine should do. We are describing what we understand about a problem — its structure, its constraints, its valid states, its transitions. The machine execution is a consequence of that description. The description itself is the intellectual artifact. The description is what took seventy years of computer science to learn how to make precise.

In Exodus, that description does not become the input to a runtime. It becomes the input to a geometry compiler — the tool that translates the logical description of an application into the geometric structure of the Vessel Image. The language still describes the logic. The logic still defines the valid states. The valid states still become the geometry. The only thing that changes is that the geometry is computed once and sampled forever, rather than re-executed on every interaction.

The language is not left behind because the language was never primarily about the machine. It was always about the problem. and the problem does not change when the execution model does.

What changes Frameworks. Runtimes. Compilers targets. Infrastructure. Everything that was aimed at making execution cheaper within the imperative paradigm.
What stays Languages. algorithms. The logical descriptions of problems that human minds developed over seventy years of computer science. The intellectual work. The understanding.
Why Because languages are representations of human thought, not representations of machine architecture. Human thought does not change when the machine changes. The precision of the representation — the years of work that went into making logic expressible — comes with us completely.
The language was always the most faithful representation of reality we could build inside the compute model. It remains that. It always will. Because it was never about the compute model. It was about reality. and reality does not change when the execution layer does.

This is why P-06b states that everything we built is correct. Not as comfort. as fact. The logical descriptions, the type systems, the algorithms, the design patterns — all of it correctly captured something true about the problem domain. The only thing that was incorrect was the assumption that the correct description had to be re-executed imperatively on every interaction. That assumption came from the hardware. It was never in the logic. Exodus removes the assumption. The logic remains.

P-02 The Closed-World assumption Invalid states are unrepresentable

The ledger defines the complete set of legal values structurally.

The CSS Ledger is a complete, static definition of every execution state the application can occupy. It does not validate values at runtime. Invalid states are not prevented — they are unrepresentable.

This is the same class of leap as null-safe type systems over null checks. The difference between preventing an invalid state and making it structurally unreachable.

Value Lattice

@property declarations constrain the value space per axis — the alphabet. CSS Ledger rules define which intersections require handling — the grammar. Rule complexity grows linearly with state variables, not exponentially. Together they form a value lattice enforced continuously by the browser's native selector engine.

/* CSS as type system — legal values defined structurally */
[data-status="pending"]  { --status-valid: true; }
[data-status="complete"] { --status-valid: true; }
[data-status="failed"]   { --status-valid: true; }

/* Everything else has no representation */
[data-status]:not(
[data-status=“pending”],
[data-status=“complete”],
[data-status=“failed”]
) { –status-valid: false; }

/* Cross-field constraint */
[data-role=“guest”][data-action=“delete”] { –action-valid: false; }

/* Transition enforcement */
[data-state=“complete”][data-prev-state=“pending”]       { –transition-valid: true; }
[data-state=“complete”]:not([data-prev-state=“pending”]) { –transition-valid: false; }

P-06 Vessel Image Bounded metric space · strings of relative bounds · a trillion states need no enumeration

a Vessel holds potential until context releases it into an instance.

The quantum enumeration of the complete valid state space produces a single object. It is not meant to be read directly. It is meant to be sampled from by context. The Vessel Image is a bounded metric space of application state — a geometric structure where every valid state is a point, every invalid state is an absence, and the distance between any two states is structural information encoded once and read without computation.

The Metric Space

a metric space defines distance between points in a way that is meaningful, consistent, and structurally encoded. In the Vessel Image, distance between states is not computed at runtime. It is measured at sample time against a static object. The cost of a state transition is the distance between its origin and destination coordinates. The validity of a transition is whether the destination coordinate exists. Neither requires execution. Both require only a GPU sample.

Point a valid application state. Exists at a coordinate. If the coordinate is empty, the state does not exist.
Distance The structural cost of transitioning between two states. Encoded in the geometry. Read by sampling. Never computed.
Boundary The edge of the valid state space. Defined by the strings and their relative bounds. Everything inside is implicitly valid. Everything outside is implicitly absent.
Curvature Where one string's valid bounds depend on another string's position, the space curves. The relativity of constraints is encoded as geometry, not as conditional logic.
Empty space Invalid states. They are not rejected — they are geometrically absent. There is no coordinate to reach. Nothing fires. The failure mode is structural impossibility, not runtime error.

Strings of Relative Bounds

The Vessel Image is a massive container of strings whose bounds are relative to each other. Each state variable is a string. Its valid values define its bounds. Those bounds are relative — the valid range of one string depends on the position of other strings in the space. a trillion combinations of states exist as the interior volume implied by boundary geometry — not as a trillion enumerated records.

String 1: role bounds: [admin, user, guest]
String 2: action bounds relative to role — admin: [read, write, delete] · user: [read, write] · guest: [read]
String 3: resource bounds relative to role + action — geometry encodes which resources each role-action pair can reach

In a flat model these combinations are enumerated explicitly. In the metric space, the relativity of bounds is encoded as curvature. The GPU reads the curvature. Valid combinations emerge from the geometry. Nothing is enumerated. The interior of the space is implicit — defined by its boundary, not listed point by point.

You do not store a trillion states. You store the shape that implies them. The shape is compact. The implications are infinite. The GPU reads the shape. The application occupies the implications.

Distance as architecture

Two states that are geometrically close are structurally similar. Transitions between them are cheap — low distance, few boundary crossings, shared execution context. Two states that are geometrically distant are structurally different. Transitions between them cross more boundary regions. The cost is structural, not computed.

This changes what debugging means. You do not read logs to understand why a transition failed. You read geometry. The failed transition attempted to reach a coordinate that does not exist, or crossed a boundary it was not permitted to cross. The geometry tells you exactly where the boundary was and exactly what the nearest valid coordinate is. Debugging becomes pathfinding.

P-06b Everything We Built Is Correct The right solution to the wrong problem · all of mankind's work comes with us

The existing compute model solved the right problem. It framed it wrong. Everything fundamental comes with us.

This is not a statement about what came before being inadequate. It is a statement about where the work was aimed.

Every language ever written solves logic. Correctly. Every compiler ever built resolves logic into executable form. Correctly. Every algorithm, every data structure, every design pattern, every framework — all of it correctly solves the problem of expressing and executing logic. The work is right. The work has always been right.

What was wrong was not the solution. What was wrong was the assumption that logic had to be executed imperatively — that solving logic meant switching transistors in sequence, branching on conditions, managing mutable state across a call stack. That assumption was never a logical necessity. It was a hardware constraint that became a cultural assumption that became invisible.

Mankind invested everything in the right solution to the wrong problem. The solution does not need to be discarded. It needs to be recontextualised. The logic was always correct. The execution layer was the wrong frame.

What Carries Forward Without Change

Every language Languages are absolute. They express logic precisely. That precision does not change. The language becomes the authoring surface for geometry — the input to the Vessel compiler rather than the input to a runtime. The logic it expresses is unchanged. Where it runs changes completely.
Every compiler Compilers resolve logic into executable form. In Exodus, the target of compilation is the geometric structure of the Vessel Image rather than imperative bytecode. The compiler's job — resolving logic into a form a machine can act on — is identical. The machine it targets is the GPU. The form it produces is geometry.
Every algorithm algorithms solve problems. They still solve them. In Exodus, algorithms run once — at enumeration time — and their results are encoded in the geometry of the Vessel. The algorithm does not run on every user interaction. It runs once. Its answer is read forever after by sampling. The algorithm is correct. It just no longer needs to run repeatedly.
Every database Databases store and retrieve structured information. The Vessel Image is a database — a higher-dimensional one whose schema is geometry rather than tables. Every principle of database design — normalisation, indexing, referential integrity, transaction boundaries — has a direct geometric equivalent. The discipline carries forward. The storage model changes.
Every framework Frameworks manage complexity. That need does not disappear. In Exodus, frameworks become geometry authoring tools — ways of expressing the state spaces and boundary conditions that the Vessel compiler turns into geometry. The abstraction is still necessary. What it abstracts changes.

The Wrong Problem That Was Being Solved

The current compute model treats application logic as a problem of execution — how do you run the right instructions in the right order for every user on every interaction? This framing made execution efficiency the central engineering concern. Faster CPUs. Better branch prediction. Cache locality. Concurrency primitives. Garbage collection. all of it aimed at making execution cheaper.

But execution is not the problem application logic needs solved. application logic is a problem of definition. What states are valid? What transitions are permitted? What constraints apply across which combinations of state? These are questions about the shape of a space — not questions about the sequence of instructions that traverse it.

The current model answered a question about traversal when the question was always about shape. Every execution optimisation in history made traversal cheaper. None of them made the shape smaller, because none of them recognised that the shape was the problem.

Sacrificial Purpose Recontextualises Without Discarding

The law of Sacrificial Purpose states that every layer is instantiated by the one below it, consuming the lower layer's full expressiveness as a single bootstrapping act. This means the current compute model — every language, compiler, algorithm, and framework — is not superseded. It is consumed. Its full expressiveness becomes the input to the next dimensionality expansion.

The web stack does not disappear. It becomes the bootstrapper. JavaScript does not disappear. It becomes the memory bus. CSS does not disappear. It becomes the translational bridge between GPU geometry and CPU output. Every tool humanity built to solve the logic problem carries forward — recontextualised as the layer that instantiates the space where the logic already lives, encoded geometrically, waiting to be sampled.

Nothing is wasted. Nothing is wrong. Everything is recontextualised. The solution was always correct. It was aimed at the wrong layer. Exodus is not the replacement of what came before. It is the thing that what came before was always in the process of instantiating.

What Changes

The only thing that changes is where logic lives and when it runs. In the current model, logic lives in execution and runs continuously — on every user interaction, for every concurrent session, for the lifetime of the application. In the Exodus model, logic lives in geometry and runs once — at enumeration time — and thereafter exists as a static object that sessions sample without re-executing.

The logic is identical. The shape it produces is identical. The difference is that the current model rediscovers the shape on every request, and the Exodus model reads the shape from an object that already knows it.

In traditional systems validation requires N separate service calls per write. In Exodus, validators are already-running continuously-evaluating state machines inside the CSS Ledger. The quorum is not triggered by a write. It is already resolved before the write is proposed.

/* Quorum as emergent cascade state */

P-14 Emergent Encryption Superposition as security · coordinates without context are every state · the key never exists

The application is defended by quantum superposition. The key that unlocks it never exists as a discrete artifact.

Conventional encryption separates data from key. The key must be stored, transmitted, managed, and trusted. Every one of those moments is an attack surface. The entire PKI infrastructure exists because the key exchange has to happen across an adversarial network.

Exodus eliminates that surface not by making transit safer — but by ensuring the meaningful content never transits as a discrete artifact in the first place.

The Coordinate System as Security Model

Every piece of application state in the Vessel Image exists at a coordinate in a multidimensional space. a coordinate without its exact context slice does not point to a specific state. It points to every possible state simultaneously — the full superposition of the enumerated state space. an attacker who obtains coordinates without the compression context has captured a position in a space that holds every possible application state at once. The coordinate is meaningless without the exact slice that collapses it.

Coordinate alone Points to superposition. Every valid state the application can occupy exists at that address simultaneously. No application-specific information is recoverable.
Compression context alone Describes a camera position into a space. Meaningless without the Vessel to sample from.
Vessel alone The complete enumerated state space. Contains everything and nothing — equivalent to possessing every possible version of the application simultaneously.
Coordinate + exact context Collapses superposition to a single valid state. This is the only combination that produces meaningful content. It exists for exactly one session, derived once, then gone.

The Network Is Killed Before the Key Exists

The context extractor derives the compression algorithm from the intersection of @media conditions, IPC state, session details, and pod lineage. This intersection is unrepeatable — no two sessions occupy the same viewport, connection quality, authentication state, and pod generation simultaneously. The compression algorithm that produces the memstrap is the encryption key. But the key is derived locally, after the Vessel has already arrived. The network carries generic potential. The meaningful, session-specific content materialises on the device, after the network has done its job and become irrelevant.

The most secure data is data that only ever exists as an emergent derivation — never as a copyable object. The Vessel can be public. The slice is inherently private because it cannot be reproduced without the exact context that generated it, and that context is already gone.

Security Properties by Construction

No key to steal The compression key is a one-time emergent property of a specific moment of context intersection. Used once, dissolved. Never stored. Cannot be intercepted — it did not exist during transit.
No secret to manage The Vessel Image is not a secret. Possessing it reveals nothing about any specific session's valid state without the unrepeatable context that collapses it.
No attack surface in transit a perfect interception of the Vessel yields the superposition of every possible application state. Equivalent to capturing a dictionary to intercept a specific word. The word was never on the wire.
Superposition as defence The quantum enumeration layer that produces the Vessel is not incidentally related to the security model — it is the security model. The property that makes complete state space enumeration tractable is the same property that makes any individual state unrecoverable without context.

Relationship to Existing Security Infrastructure

This is not a replacement for transport security, authentication, or the Chungus validation layer. It is a security property that emerges from the architecture without being designed in. TLS still secures the WebSocket. authentication still governs session initiation. Chungus still gates database writes. What Emergent Encryption adds is a guarantee that even a fully compromised transport, a stolen Vessel, and a captured coordinate set yields nothing recoverable — because the collapse from superposition to specific state requires a context that existed once, was used once, and is gone.

The Vessel is public. The coordinates are public. The context is unrepeatable. The slice exists once. Security through the physics of the model — not through the addition of cryptographic infrastructure.

P-15 Multi-Runtime Composition applications are not integrated · they are composed · the integration layer ceases to exist

Multiple runtimes pack into one object. Integration is not a layer between systems — it is the geometry between regions of the same Vessel.

Today software integration is additive. You build application a. You build application B. You write the integration layer between them. Three separate things that have to stay in sync as each evolves independently. Every integration is a bilateral contract negotiated across a network boundary — REST endpoints, webhooks, Oauth handshakes, schema mapping, version management, error handling. The boundary is structural. The friction is permanent.

In the Vessel model, that boundary does not exist. Integration is not a third thing you build. It is the geometry between two regions of the same space.

The Fourth axis

The Vessel Image currently encodes state across three spatial axes. add a fourth:

X axis State variable
Y axis Valid values along each state dimension
Z axis Relationship depth — parent, child, sibling context
W axis Runtime identity — which application occupies this region

a coordinate now describes a state within a specific runtime within the unified object. The geometry between runtimes — the space between different W-axis regions — encodes how they interact, what state can pass between them, and what the valid transition paths are across runtime boundaries.

What Happens to the Integration Layer

Two runtimes in the same Vessel do not have an aPI boundary. They have a geometric relationship. State that can pass from runtime a to runtime B exists as coordinates in the space between their W-axis regions. Valid integrations are geometrically adjacent. Invalid integrations are geometrically absent — empty space. There is no aPI to call because there is no boundary to cross. There is a coordinate to move to.

Integration bugs Geometrically impossible states. They do not exist in the object. They cannot be reached.
Version mismatches Coordinate drift. Detectable as geometric discontinuity in the Vessel diff — visible before deployment.
Schema changes Region boundary shifts. The diff of two Vessel versions shows exactly what moved, what is now unreachable, and what new coordinates appeared.
aPI contract failures Traversal into empty space. No execution surface exists at that coordinate. Nothing fires. The failure mode is structural absence, not runtime error.
Integration layer Does not exist. Never existed. Was always a workaround for the absence of shared geometry.

Software Composition Becomes Geometric

You do not describe every possible way two applications can interact. You define the shape of each application's state space and the valid transitions at their boundaries. The quantum enumeration layer produces a single Vessel that encodes both regions and the geometry between them simultaneously. The context extractor renders only the coordinates relevant to the current session — which may span both runtimes or neither.

applications are not integrated. They are composed. The difference is the difference between negotiating a treaty and sharing a coordinate system.

What This Means for Enterprise Software Suites

any product suite — creative tools, enterprise platforms, productivity applications — currently maintains integration layers between products that have accumulated years of friction, version drift, and schema mismatches. These layers exist because each product is a separate runtime with a separate state space and no shared geometry.

a composed Vessel encodes each product as a region on the W axis. Moving an asset, a document, a user record, or a workflow between products is not an export, an import, or an aPI call. It is a coordinate transition within the same object. The context extractor renders the slice relevant to whichever region the session is currently occupying. The Chungus validation layer ensures that transitions between regions respect the geometric constraints — that state arriving in one runtime from another is valid in its new coordinate context.

The Deeper Structural Claim

The reason multiple runtimes can pack into one object is the same reason the Vessel Image resolves combinatorial explosion. Geometry absorbs complexity that enumeration cannot. Two runtimes integrated via aPI require explicit enumeration of every possible interaction. Two runtimes integrated via geometry require only the definition of their regions — the space between them encodes the interactions implicitly, completely, and without a line of integration code.

The integration layer is not simplified. It is not abstracted. It does not become a managed service. It ceases to exist as a category. What replaced it was always there — it just had no name and no address until the coordinate system made it visible.

S-01 The Gate Mechanism Core primitive · CSS schedules JS
Implements P-01 Role assignment — JavaScript as memory bus, CSS as executor

The browser's style engine schedules logic. Not developer code.

a registered CSS custom property is assigned conditionally by a selector. When selector state changes, a zero-duration transition fires. transitionend delivers the signal to JavaScript. JS does not schedule itself — the CSS Ledger does.

@property --gate-open {
  syntax: '<number>';
  initial-value: 0;
  inherits: false;
}

.order-node[data-items][data-address][data-payment] {
–gate-open: 1;
transition: –gate-open 0s;
}

node.addEventListener(‘transitionend’, (e) => {
if (e.propertyName !== ’–gate-open’) return;
const current = getComputedStyle(node)
.getPropertyValue(’–gate-open’).trim();
const prev = node.dataset.prevGateOpen;
if (current === prev) return;
node.dataset.prevGateOpen = current;
// JS executes here. Precondition guaranteed by CSS.
});

Implementation Rules

  • @property registration is mandatory
  • syntax must be ’’
  • initial-value must be 0
  • inherits must be false
  • transition duration must be 0s
  • Both transitionend and webkitTransitionEnd must be registered (Safari)
S-02 The DOM as State Store No other state store exists
Implements P-01 Role assignment — DOM as attested runtime

The DOM stores all committed state. There is no other state store.

<div class="ledger-node order-node"
  data-id="order-001"
  data-items="3"
  data-address="confirmed"
  data-payment="card"
  data-prev-gate-open="0"
></div>

attribute Categories

Identity data-id — never used in CSS Ledger selectors. Stable identifier only.
State data-[key] — present when value exists, absent when it does not. Written only via Batch Write.
Snapshots data-prev-[gate] — written only by JS reconciliation in the Write Guard.

absence of a value is removeattribute, not empty string. [data-amount] matches presence. an attribute set to "" also satisfies the selector. These are not equivalent — treat them as distinct.

S-03 The Dual-Signal Model No gate change permanently missed · 16ms maximum latency
Implements P-03 Resilience — implementation-layer reliability guarantee

transitionend paired with a targeted raF sweep.

transitionend is not guaranteed in all browser conditions. The dual-signal model ensures no gate change is ever permanently missed.

Fast path transitionend fires immediately when the browser processes the gate change. Zero scheduling latency.
Fallback path One-shot requestanimationFrame armed on every batch commit. Fires once on the next frame. Not a polling loop.

Both paths call the same reconcile function. The snapshot comparison inside reconcile makes it idempotent — if the fast path already ran, the fallback exits immediately. One execution per gate change regardless of how many signals fire.

Maximum latency on a missed transitionend: one frame (~16ms at 60fps).

S-04 The Write Guard Preventing re-entrant reconciliation loops
Implements P-01 Role assignment — containing JS to its memory bus role

Snapshot writes during reconciliation must not re-enter JS.

When JS writes a snapshot attribute during reconciliation, that write triggers CSS Ledger re-evaluation, which may fire another transition, which re-enters JS. The write guard prevents this loop.

#reconcile(propertyName) {
  if (this.#flushing) return;
  this.#flushing = true;

const current = getComputedStyle(this.node)
.getPropertyValue(propertyName).trim();
const prev = this.node.dataset[snapshotKey];

if (current !== prev) {
this.node.dataset[snapshotKey] = current;
this.#dispatch(propertyName, current, prev);
}

requestanimationFrame(() => { this.#flushing = false; });
}

Guard is set at reconciliation start, released at the next frame boundary. Re-entrant calls within that window are blocked. Events that arrive during the flush are caught by the raF fallback on the following frame. Guard is per-node — separate nodes have independent flush states.

S-05 Data Normalisation DOM never holds an unnormalised value · runtime enforcement of closed-world
Implements P-02 Closed-World assumption — runtime enforcement at write time

all attribute values are normalised before they reach the DOM.

This prevents comparison bugs, false deltas in reconciliation, and CSS Ledger selector inconsistencies. This is the runtime enforcement of what @property declarations define structurally — both layers constrain the same value space.

const registry = {
  amount:    { type: 'float',  precision: 2 },
  recipient: { type: 'string', transform: 'lowercase' },
  method:    { type: 'enum',   values: ['card', 'cash', 'crypto'] },
  quantity:  { type: 'int' },
  active:    { type: 'bool' },
};

a null result removes the attribute. It does not set it to "" or the string "null". Normalisation runs during batch set(), not during commit().

Without '45.0' !== '45' — false delta, false gate event, false execution.
With Both become '45.00' — no false gate events.
S-06 Batch Write System One recalculation per logical write · no micro-cascade storms
Implements P-01 Role assignment — atomic state transitions in the DOM

Multiple attribute writes committed in one synchronous burst.

Without batching, intermediate selector states may produce transition events for partial state — micro-cascade storms. The batch system prevents this by exploiting the browser's rendering model: a synchronous burst of setattribute calls within a single task produces exactly one style recalculation.

ledger.batch()
  .set('items', 3)
  .set('address', 'confirmed')
  .set('payment', 'card')
  .commit();
// One recalculation. One potential gate event.

The raF fallback is armed immediately after every commit(). Direct setattribute calls on state attributes in application code are prohibited. all state writes must go through the batch interface.

S-07 The Ledger Node Contract The complete unit of Exodus execution at component scale
Implements all Principles — the atom of the CSSLedger system

a ledger node is a purpose-built state carrier.

It is the concrete instantiation of the Exodus execution model at component scale. a ledger node is not a UI component. It does not own rendering or manage children. Not every element in an application is a ledger node.

LedgerNode.init(element, {
  type: 'order-node',
  registry: {
    items:   { type: 'int' },
    address: { type: 'enum', values: ['pending', 'confirmed'] },
    payment: { type: 'enum', values: ['card', 'cash', 'crypto'] },
  },
  gates: ['--gate-open'],
  onGate: (propertyName, current, prev, node) => {
    /* application logic lives here and only here */
  },
});

The type registry enforces the closed-world value constraint at write time. The gates list defines which @property registers this node monitors. onGate is the application's only valid entry point for logic.

S-08 The Four Problem Spaces Structural advantage over conventional approaches · testable claims
Implements P-02, P-03, P-04 — demonstrating where the model outperforms

1. Layout-Dependent Logic

Container queries resolve synchronously with style recalculation. ResizeObserver fires after layout. CSSLedger exposes container query resolution as a gate signal. More deterministic and earlier in the rendering pipeline than ResizeObserver-based approaches.

2. Hydration State Mismatch

State IS the DOM. Server renders attribute state. CSS Ledger evaluates gates from it on load. JS attaches and the system is live. No state recomputation pass. Hydration state recomputation is eliminated.

3. Temporal State Without JS Timers

CSS animations pause on backgrounded tabs, respect prefers-reduced-motion, do not drift, require no manual cleanup. animationend acts as a gate signal. Lifecycle-aligned timing that replaces most UI-layer timers.

4. Pseudo-Class State as application Signals

:focus-within, :valid, :invalid, :hover resolve synchronously inside the style engine. CSSLedger surfaces them as gate events. Zero-bookkeeping UI state signals with no event delegation or listener trees.

S-09 The Ledger Graph Multi-node coordination · cycle detection · shared signals
Implements P-05 Shared Memory — multi-node coordination model

Nodes coordinate through defined relationship types.

Parent/child Child writes a summary signal to parent's state attributes via batch write.
Peer dependency One node's reconciliation writes to another node's state attributes.
Shared signal Dedicated signal node holds application-level signals any ledger node may read from.

Cycle Detection

Cycle detection is mandatory. a frame-scoped cascade log tracks which nodes have been written to in the current cascade. any attempt to write to an already-written node is blocked and logged. Log clears at the frame boundary.

Cross-node writes must go through the batch system. Snapshot attributes on other nodes must never be written by outside nodes — only state attributes.
